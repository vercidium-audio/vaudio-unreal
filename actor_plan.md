# VA Actor Split Plan

## Instructions for whoever picks this up

Do **one** unchecked item below, then mark it `[x]` before ending the session. Don't do more than one unless the item is trivially small and says so. Each item should compile before being checked off.

Read `CLAUDE.md` in this plugin root first (coding guidelines: `VALog()` not raw logging, camelCase variables, no single-capital-letter acronyms, don't add unexplained null checks).

## Background

Today there is a single actor, `AVAudioEmitter` (`Source/vaudio-unreal/Public/VAudioEmitter.h` + `Private/VAudioEmitter.cpp`), that does four jobs depending on the `bIsMainListener` flag and whether `SourceSound` is set:

1. **Main listener** (`bIsMainListener = true`) — the world's single reference point. Owns `ListenerReverbSubmix`, `AmbientSound`, `TargetEmitters`, drives `ApplyListenerReverb()`, `ApplyGroupedEAXReverb()`, `ApplyAmbientFilter()`, and per-target LPF application in `Tick()`.
2. **Source** (`bIsMainListener = false`, has `SourceSound`) — plays a 3D sound, raytraced/muffled against the listener via `TargetEmitters`, sends to grouped EAX submix (`UpdateSourceSubmix()`), gets per-target LPF applied by the listener's `Tick()`.
3. *(Not yet split out)* **Relative sound** — should reuse the listener's reverb without being raytraced.
4. *(Not yet split out)* **Ambient sound** — should reuse the listener's ambient filter without being raytraced or having reverb.

Note: `AVAudioWorld::Tick()` (`Source/vaudio-unreal/Private/VAudioWorld.cpp:226`) already has a debug message that says *"remove it and use VASourceRelative actors instead"* — the split was anticipated but never implemented. `CLAUDE.md` also already lists `VAudioRelativeSource.cpp` and `VAudioAmbientSource.cpp` as TODOs.

Target end state — **five** actor classes. The key distinction the original 4-actor plan missed: "is raytraced against the listener" and "plays a sound" are independent axes. `AVAudioSource` conflated them (raytracing target that also plays a sound). `AVAudioContinuous` is the missing piece: a pure raytracing target that plays nothing, for actors (e.g. an enemy) that need an up-to-date muffle/occlusion value every frame so that many one-shot sounds attached to them (footsteps, voice barks) can reuse it via `AVAudioRelativeSource` instead of each one raytracing independently.

| Actor | Raytraced? | Reverb? | Plays sound? | Is the listener? |
|---|---|---|---|---|
| `AVAudioListener` | N/A (is the reference point) | Owns listener reverb submix | No | Yes, exactly one per world |
| `AVAudioSource` | Yes (occlusion/permeation vs. listener) | Via grouped EAX submix (directional/positional) | Yes, 3D | No |
| `AVAudioContinuous` | Yes (occlusion/permeation vs. listener) | Via grouped EAX submix (directional/positional) | No | No |
| `AVAudioRelativeSource` | No | Reuses reverb/muffling from either the `AVAudioListener` (footsteps on the player) or an `AVAudioContinuous` (footsteps on an enemy), non-directional | Yes, 2D/attached-to-source-actor | No |
| `AVAudioAmbientSource` | No | None | Yes, 2D | No |

`AVAudioSource` and `AVAudioContinuous` are nearly identical — both are raytracing targets with occlusion/permeation and grouped-EAX reverb — the only difference is whether they own a `SourceSound`/audio component. Item 3 below factors their shared raytracing-target behaviour so item 3b (`AVAudioContinuous`) is a small diff on top, not a re-implementation. Consider whether `AVAudioContinuous` should actually be `AVAudioSource`'s base class (source *is-a* continuous-that-also-plays-a-sound) rather than a sibling — decide when you get to item 3 and note the reasoning.

`AVAudioWorld` (`VAudioWorld.h`/`.cpp`) stays as-is conceptually but its `RegisteredEmitters` / `GetMainListener()` / `RegisterEmitter()` bookkeeping needs to accept the new types (see item 1). `TargetEmitters` (the list the listener raytraces against) must be able to hold both `AVAudioSource*` and `AVAudioContinuous*` — see item 2.

## Design decisions already made (don't re-litigate)

- **Shared base class**: introduce `AVAudioEmitterBase` (or similar) that owns only what's truly shared: `AudioWorld` pointer, `VAEmitter* Emitter`, `EmitterIndex`, `GetVAEmitter()`, `TryInitializeEmitter()`-style lifecycle (`BeginPlay`/`EndPlay`/`Tick` skeleton), and the low-level `vaEmitterCreate`/`vaWorldAddEmitter`/`vaEmitterDestroy` plumbing. Every VA actor still needs a `VAEmitter*` under the hood (confirm against `vaudio.h` — raytracing targets and ambient filters are computed per-`VAEmitter`), so this base class is not optional scaffolding, it reflects real shared SDK state. **Done in item 1.**
- **Raytracing-target behaviour is shared between `AVAudioSource` and `AVAudioContinuous`.** Both are things the listener raytraces against (occlusion/permeation UPROPERTYs, grouped/directional EAX submix, registration into the listener's `TargetEmitters`). The only difference is that `AVAudioSource` also owns a `SourceSound`/audio component and `AVAudioContinuous` plays nothing. Factor this shared piece once (either a common intermediate base, e.g. `AVAudioRaytracedEmitterBase`, or by making `AVAudioContinuous` the parent and `AVAudioSource` the child that adds sound-playback — pick when you reach item 3 and note the reasoning in a comment) rather than duplicating occlusion/permeation/grouped-EAX code across the two classes.
- **Relative and Ambient sources do not raytrace** — they should NOT create occlusion/permeation ray settings (`OcclusionRayCount` etc.) at all; those UPROPERTYs should not exist on those classes. Only `AVAudioSource`, `AVAudioContinuous`, and `AVAudioListener` deal with ray counts.
- **Relative source reverb/muffling has two modes**, chosen by which actor it's attached to/reused from:
  - Attached to the `AVAudioListener` (e.g. player footsteps) — reuse the listener's own non-directional reverb, same mechanism as today's `ApplyListenerReverb()` (from `vaEmitterGetEAX()` on the listener).
  - Attached to an `AVAudioContinuous` (e.g. an enemy's footsteps) — reuse that continuous emitter's already-computed muffling/EAX result (the same per-frame raytrace result `AVAudioSource` would use for itself), so many relative sounds on one enemy share a single raytrace instead of each running its own.
  Expose this as a single reference (e.g. `AVAudioEmitterBase* ReverbSource`) that accepts either an `AVAudioListener` or `AVAudioContinuous`, rather than two separate UPROPERTYs — see item 4.
- **Ambient source filter** = same mechanism as today's `ApplyAmbientFilter()`, driven by `vaEmitterGetAmbientFilter()` on the listener's `VAEmitter`. Multiple ambient actors should eventually be supported (today's `AmbientSound` on the listener is marked "WIP - only supports one ambient sound" — fixing that is a nice side effect of giving ambient its own actor class, but is NOT in scope unless a checklist item says so explicitly).
- **No backwards-compat shims.** `AVAudioEmitter` is being deleted once its pieces are migrated (see final item). Don't keep it around "just in case" or add deprecated wrappers.
- **Blueprint breakage is expected and acceptable.** This plugin is still pre-1.0-stable internal tooling (see recent commits). Existing placed `AVAudioEmitter` actors in levels will need to be manually replaced with the new actor types — note this in the PR/commit message when the final item lands, but do not attempt automatic migration tooling unless asked.

## Checklist

- [x] **1. Create `AVAudioEmitterBase`** (`Public/VAudioEmitterBase.h`, `Private/VAudioEmitterBase.cpp`) as an abstract-ish common base for all four actor types (existing `AVAudioEmitter` will inherit from it temporarily until item 6 deletes it). Move into it: `AudioWorld` UPROPERTY, `Emitter`/`GetVAEmitter()`, `EmitterIndex`/`GetEmitterIndex()`/`SetEmitterIndex()`, the position-sync-to-`vaEmitterSetPosition` logic from `Tick()`, and the common parts of `BeginPlay`/`EndPlay` (create/destroy `VAEmitter*`, add/remove from `vaWorld`, register/unregister with `AudioWorld`). Leave a pure-virtual or protected hook (e.g. `virtual bool InitializeTypeSpecific()`) for subclasses to fill in their own setup. Update `AVAudioWorld::RegisteredEmitters` (currently `TArray<AVAudioEmitter*>`, `VAudioWorld.h:201`) and `RegisterEmitter`/`UnregisterEmitter`/`GetMainListener()` signatures to use `AVAudioEmitterBase*` instead of `AVAudioEmitter*`. This is the foundation item — do this first, before any of items 2-5.

- [x] **2. Create `AVAudioListener`** (`Public/VAudioListener.h`, `Private/VAudioListener.cpp`), inheriting `AVAudioEmitterBase`. Move from `AVAudioEmitter`: `bAutoFollowCamera`, `ListenerReverbSubmix`, `AmbientSound`, `bAmbientThroughReverb`, `TargetEmitters` (retype to `TArray<AVAudioEmitterBase*>` — it must hold both `AVAudioSource*` and `AVAudioContinuous*` once items 3/3b exist, since both are raytracing targets; until then leave a TODO comment noting the intended element type), all `Reverb`/`Muffling`/`Ambient` category UPROPERTYs (these configure the listener's own raytracing, since it's the listener that raytraces *toward* targets), `RelativeReverbInnerThreshold`/`OuterThreshold`, `ApplyListenerReverb()`, `ApplyGroupedEAXReverb()`, `ApplyAmbientFilter()`, the target-registration loop and per-target LPF-push loop from `Tick()` (`VAudioEmitter.cpp:262-337`), `ListenerReverbPreset`, `AmbientAudioComponent`. Call `vaEmitterSetHasRelativeReverb(Emitter, true)` unconditionally (was conditional on `bIsMainListener`). There is no `bIsMainListener` flag on this class — its existence as an `AVAudioListener` instance IS the flag. `AVAudioWorld::GetMainListener()` should return the first registered `AVAudioListener` (keep the existing "first one wins, warn on duplicates" behavior from `RegisterEmitter()` at `VAudioWorld.cpp:429`).

- [x] **3. Create `AVAudioSource` and `AVAudioContinuous`** together, since they share all raytracing-target behaviour and only differ in whether they play a sound (see "Design decisions" above for how to factor the shared piece — decide the class relationship here and note it in a comment).
  - **Shared raytracing-target behaviour** (`Public/VAudioSource.h`/`Private/VAudioSource.cpp` and/or a common intermediate, e.g. `VAudioContinuous.h`/`.cpp` as the base): `bAffectsGroupedEAX`, `Reverb`/`Muffling`/`Ambient` category UPROPERTYs that describe *this emitter's own* ray behavior as a raytracing target (re-check against `vaudio.h` which of `ReverbRayCount` etc. apply per-source vs. only make sense on the listener — the current code sets all of them on every emitter regardless of role, so verify against SDK semantics rather than assuming), `CurrentGroupedEAXIndex`, registration into the listener's `TargetEmitters`, whatever produces the per-target grouped-EAX/muffling result each tick (today spread across `UpdateSourceSubmix()` and the listener's per-target loop).
  - **`AVAudioContinuous`** (`Public/VAudioContinuous.h`, `Private/VAudioContinuous.cpp`): just the shared raytracing-target piece above, no `SourceSound`/audio component/playback of any kind. Used for actors (e.g. an enemy) that need a continuously up-to-date muffle/occlusion value so that multiple one-shot `AVAudioRelativeSource`s attached to them (footsteps, barks) can reuse a single raytrace instead of each raytracing independently. Expose whatever accessor `AVAudioRelativeSource` will need in item 4 (e.g. `GetMufflingResult()`/`GetGroupedEAXIndex()` — match to whatever the shared piece actually computes).
  - **`AVAudioSource`** (`Public/VAudioSource.h`, `Private/VAudioSource.cpp`, inheriting the shared piece / `AVAudioContinuous` per the decision above): adds `SourceSound`, `bLooping`, `MaxVolume`, `SourceAudioComponent`, `SourceLPFPreset`, `SourceEffectChain`, `bSourcePendingSpawn`, `ApplySourceFilter()`, `SetDryOutputEnabled()`, `UpdateSourceSubmix()`, `TrySpawnSourceSound()`. This is the direct equivalent of today's "emitter with `SourceSound` and `bIsMainListener = false`" path.

- [ ] **4. Create `AVAudioRelativeSource`** (`Public/VAudioRelativeSource.h`, `Private/VAudioRelativeSource.cpp`), inheriting `AVAudioEmitterBase`. For footsteps/gunshots/one-shot sounds relative to either the listener or an `AVAudioContinuous`. NOT raytraced — do not expose `OcclusionRayCount`/`PermeationRayCount`/etc, and do not add this actor to any listener's `TargetEmitters`. Expose a single reference UPROPERTY (e.g. `AVAudioEmitterBase* ReverbSource`, validated at `BeginPlay`/in-editor to only accept an `AVAudioListener` or `AVAudioContinuous` — warn clearly per `CLAUDE.md` if misconfigured) rather than two separate fields:
  - If `ReverbSource` is an `AVAudioListener`: reuse that listener's own non-directional reverb, same mechanism as today's `ApplyListenerReverb()` (from `vaEmitterGetEAX()` on the listener).
  - If `ReverbSource` is an `AVAudioContinuous`: reuse that continuous emitter's already-computed grouped-EAX/muffling result from item 3b, so many relative sounds on one enemy share its single raytrace.
  On `BeginPlay`/spawn, play `SourceSound` (2D or attached-to-source-actor, matching today's `SpawnSound2D` pattern used for `AmbientSound` in `VAudioEmitter.cpp:120`) and route it through whichever submix/preset the `ReverbSource` resolves to above. No occlusion/permeation UPROPERTYs at all — this is the main functional difference from `AVAudioSource`/`AVAudioContinuous`. Confirm with `vaudio.h` whether a relative source still needs its own `VAEmitter*` at all (it needs *a* position for potential future directional panning, but since it skips raytracing entirely, check whether `vaEmitterCreate()`/`vaWorldAddEmitter()` are even necessary, or whether this can skip the VA SDK entirely and just be a thin UE audio component wrapper reading the reverb source's current preset/muffling). This determination affects whether it inherits `AVAudioEmitterBase` or is a lighter-weight standalone actor — decide and note the reasoning in a comment.

- [ ] **5. Create `AVAudioAmbientSource`** (`Public/VAudioAmbientSource.h`, `Private/VAudioAmbientSource.cpp`). For rain/wind/room-tone sounds. NOT raytraced, NO reverb. Plays `SourceSound` in 2D and applies a low-pass filter driven by the main listener's `vaEmitterGetAmbientFilter()` result (same math as today's `ApplyAmbientFilter()` in `VAudioEmitter.cpp:374`), polled from `Tick()` via `AudioWorld->GetMainListener()->GetVAEmitter()`. Like item 4, confirm whether this needs its own `VAEmitter*`/SDK registration or can just read the listener's already-computed ambient filter each tick without registering itself as an SDK emitter at all — it has no occlusion/permeation/reverb settings of its own, so it may not need one. This class is also the natural place to finally support *multiple* ambient sounds (today's single `AmbientSound` on the listener is marked WIP-single-sound) since each instance is now independent — do this if it falls out naturally, but don't gold-plate it if it doesn't.
